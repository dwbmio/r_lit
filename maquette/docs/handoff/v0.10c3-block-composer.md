# v0.10 C-3 — Block Composer (second window) + Save Local Draft + Publish to Hfrog

**Shipped 2026-04-28 (afternoon).** Closes the loop opened by C-2:
the user can now *create* new blocks rather than just consume them.
A standalone "Block Composer" window pairs a 3-D block preview with
a multi-attempt prompt console; the result is either a local draft
(`~/.cache/maquette/blocks/local-drafts/<id>.{json,png}`) or an
artifact published straight to hfrog.

Per the v0.10 C-2 hand-off, this commit lights up the
`BlockMetaSource::LocalDraft` variant and the
`HfrogPublisher` write-side client that were both staged in the
data layer last commit.

## §1 — Layout (matches the brief)

```
┌─ Maquette · Block Composer ─────────────────────────────────────┐
│                                                                  │
│  ╭ Shape ╮                  Block Composer                       │
│  │ Cube  │  ← floating       (multi-line prompt)                 │
│  │ Sphere│                   Provider: [rustyme · cpu ▼]          │
│  ╰───────╯                   Style:    [auto (default) ▼]         │
│                              Seed: [1] ↻   Size: [128]²           │
│   ┌──────────┐               [Generate]                           │
│   │   3-D    │               ─── History ───                      │
│   │ preview  │               #3 fal · seed 5 · 256×256            │
│   │  block   │                  "weathered iron, scratched..."     │
│   │ (cube /  │                  [✔ Selected] [Replay] [Discard]   │
│   │  sphere) │               #2 cpu · seed 3 · 128×128            │
│   │ + texture│               …                                     │
│   └──────────┘               ─── Save / Publish ───              │
│                              id:           [iron_block]           │
│                              name:         [...]                  │
│                              description:  [...]                  │
│                              texture_hint: [...]                  │
│                              tags:         [metal,industrial]     │
│                              [Save Local Draft] [Publish to Hfrog]│
└──────────────────────────────────────────────────────────────────┘
```

* **Left half** — a Bevy 3-D scene rendered into the second OS
  window via a dedicated `Camera3d` + `RenderTarget::Window`. A
  single mesh entity carries `BlockComposerPreviewMesh`; switching
  shape swaps the `Mesh3d` handle, switching attempts swaps the
  material's `base_color_texture`. PanOrbit on the camera so the
  user can rotate the block.
* **Floating shape picker** — `egui::Area` anchored top-left on
  the second window; Cube / Sphere selectable_label.
* **Right SidePanel** — the prompt + history + draft form +
  action buttons. All managed by `composer_ui_system` running on
  a per-window egui schedule (`ComposerContextPass`).
* **Status bar** — bottom of right panel summarises which file
  paths Save / Publish hit.

The window itself is non-modal: opening / closing it doesn't
disturb the main editor's state. Closing is a "minimise to nothing"
— the in-flight conversation isn't preserved across a close.

## §2 — Module map

### Lib additions (`maquette/src/block_meta.rs`)

* `BlockMetaSource::LocalDraft { created_at: i64 }` — new variant.
  `is_empty` / `label` / serde all updated; v0.10 C-2 files
  load unchanged because the `serde(tag = "kind")` enum encoding
  rejects unknown variants by default → the `LocalDraft` payload
  is wire-compatible with the existing schema.
* `pub mod drafts` — new submodule with:
  * `LocalDraftProvider` (impl `BlockMetaProvider`).
  * `write_draft(cache_dir, &meta, png_bytes) -> PathBuf` — atomic
    JSON + sibling PNG.
  * `remove_draft` (best-effort idempotent delete).
  * `read_draft_png` (read-only helper).
  * `merge_into_library(existing) -> Vec<BlockMeta>` — drafts
    win on id collisions so editing a draft of `grass` shadows
    the bundled `grass`.
* `pub struct hfrog::HfrogPublisher` + `pub struct PublishOutcome`:
  * `publish_block(meta, png_bytes)` does
    1. `PUT /api/artifactory/add` with the JSON ArtifactoryModel,
       gets back a presigned S3 URL,
    2. `PUT <presigned URL>` with `content-type: image/png` and
       the raw bytes,
    3. `GET /api/artifactory/find` to look up the assigned `pid`
       for the toast.
  * Inlined a tiny RFC 1321 MD5 implementation (no extra dep);
    pinned against the standard reference vectors.
  * Configurable through env: `MAQUETTE_HFROG_BASE_URL`,
    `MAQUETTE_HFROG_RUNTIME`, `MAQUETTE_HFROG_TIMEOUT_SECS`,
    `MAQUETTE_HFROG_S3_INC_ID` (defaults to 0; ops can pin a
    specific S3 bucket).

### GUI additions (`maquette/src/block_composer.rs`)

A single new module, ~1400 lines (incl. UI). Owns:

* `BlockComposerPlugin` — registers state resource, eight
  messages, twelve update systems, plus the per-window
  `ComposerContextPass` schedule.
* `BlockComposerState` — composer-only resource. Carries the
  shape, form, history (one `ComposerAttempt` per generation),
  selected attempt, async-task busy flags, last error string,
  and the editable draft form.
* Components: `BlockComposerWindow`, `BlockComposerCamera`,
  `BlockComposerPreviewMesh`, `BlockComposerLight`.
* Three async-task wrappers tied to background tasks via
  `AsyncComputeTaskPool`: `PendingGenerate`, `PendingSave`,
  `PendingPublish`. Each uses the same poll-once-per-frame
  pattern the export job uses.

### Other touchpoints

* `maquette/src/block_library.rs::source_label_color` learns
  `LocalDraft → orange` so the main library UI badges drafts
  distinctly.
* `maquette/src/main.rs` registers the new plugin alongside
  `BlockLibraryPlugin`.
* `maquette/src/ui.rs` adds a top-bar `Window` menu with `New
  Block Composer…` and threads `OpenBlockComposer` through
  `UiMessages`.

## §3 — Multi-window egui — the bevy_egui 0.39 trick

bevy_egui 0.39 only auto-runs UI systems for the primary egui
context. For the composer we declare our own schedule label and
attach an `EguiMultipassSchedule` component to the second
camera:

```rust
#[derive(ScheduleLabel, Clone, Debug, PartialEq, Eq, Hash)]
pub struct ComposerContextPass;

commands.spawn((
    Camera3d::default(),
    RenderTarget::Window(WindowRef::Entity(window)),
    EguiMultipassSchedule::new(ComposerContextPass),
    BlockComposerCamera,
));

app.add_systems(ComposerContextPass, composer_ui_system);
```

Inside `composer_ui_system` we query `&mut EguiContext` filtered
by `With<BlockComposerWindow>` — there's exactly one match while
the window is alive, zero otherwise.

Documented in detail in `bevy_egui-0.39/src/lib.rs:259..295`
(the multi-pass example block) and in this file's module-level
doc.

## §4 — Async task lifecycle

Three tasks, identical pattern, all going through
`bevy::tasks::AsyncComputeTaskPool` (no tokio):

| Task | Trigger | Backing work | Polling system |
|---|---|---|---|
| Generate | `ComposerGenerate` message | `TextureProvider::generate(&request)` (Mock or Rustyme) | `poll_generate_tasks` |
| Save Local Draft | `ComposerSaveDraft` | `block_meta::drafts::write_draft` | `poll_save_tasks` |
| Publish to Hfrog | `ComposerPublish` | `block_meta::hfrog::HfrogPublisher::publish_block` | `poll_publish_tasks` |

Concurrency rules:

* Only one task of any kind in flight at a time. UI buttons grey
  out and surface the reason on `disabled_hover_text`.
* Each polling system flips the busy flag (`generating` /
  `saving` / `publishing`) in tandem with task completion.
* Failures land on `state.error` and a red toast. The history
  entry is *not* added on a failed generate — only successful
  PNG bytes get a slot.
* Save Local Draft on success **also** triggers
  `LocalDraftProvider::merge_into_library` against the live
  `BlockLibraryState.blocks`, so the main window's library card
  list updates in the same frame without a manual Sync.

## §5 — Form / draft semantics

The right panel is a single `egui::ScrollArea`. Inside it:

### Prompt + provider form

* Multi-line `prompt` field — required, defaults empty.
* `Provider` combobox: `rustyme · cpu` / `rustyme · fal` / `mock
  (offline)`. Mock uses `MockProvider::MODEL_ID` + the request
  hash → byte-identical PNGs offline (great when the user's
  hfrog/sonargrid backplane is down).
* `Style mode` combobox shown only for cpu lane:
  `auto / solid / smart / (omit kwarg)`. Maps to
  `RustymeConfig::style_mode`.
* `Seed` text + `↻` bump button.
* `Size` text — single number (always square; clamped 16..=1024).
* `Generate` button — disabled when the prompt is empty *or*
  another async task is in flight.

### History timeline

Reverse chronological. Each card shows:

* Attempt id (monotonically counted within the session).
* Provider · seed · `WxH` summary.
* Truncated prompt (first 110 chars, italic).
* Three buttons:
  * `Select` (or `✔ Selected` when active) — applies the
    attempt's PNG as the preview block's
    `base_color_texture`.
  * `Replay` — bumps the seed, copies prompt/provider/style/size
    into the form, fires another `Generate`. Lets the user A/B
    a prompt without retyping it.
  * `Discard` — drops the entry. If it was the selected one,
    the preview reverts to the solid grey material; if it was
    earlier, the selected index shifts.

PNG thumbnails are deferred to a follow-up commit (egui's
texture loader needs a one-shot `add_image` call per attempt; we
already produce a `Handle<Image>` for the 3-D preview, but
streaming that into egui is an extra hop). For now the timeline
is text-only — adequate for "I want #3 over #5", clearer than
trying to read a 32 px thumbnail.

### Save / Publish

* `id` — required, validated for `[a-z0-9_]`.
* `name` — defaults to `id` if empty.
* `description` — multi-line.
* `texture_hint` — defaults to the selected attempt's prompt.
* `tags` — single line, comma-separated, trimmed.

Both buttons are gated on `(has_selection && id_valid && !busy)`.
Save Local Draft writes to disk and then merges into
`BlockLibraryState`. Publish to Hfrog runs the publisher; on
success it removes the matching local draft (so duplicates
don't sit around).

## §6 — Tests (5 new in lib)

In `block_meta::drafts::tests`:

* `write_then_list_round_trip` — writes a draft via
  `write_draft`, lists via `LocalDraftProvider`, checks both id
  + count.
* `remove_draft_drops_both_files` — write + remove, then verify
  list is empty *and* the sibling PNG was deleted (catching the
  half-removal race we'd hit on a partial publish).
* `remove_missing_draft_is_noop` — calling remove on a draft
  that never existed is not an error.
* `merge_drafts_override_collisions` — the bundled `grass`
  block + a draft `grass` ends up with the draft winning, and
  carries a `LocalDraft` source.

In `block_meta::hfrog::publisher_tests`:

* `md5_matches_known_vectors` — RFC 1321 reference vectors
  through the inlined MD5 (`""`, `"a"`, `"abc"`,
  `"message digest"`, `"abcdefghijklmnopqrstuvwxyz"`).

## §9 — What's deliberately deferred

* **PNG thumbnails in the History list.** egui texture-loading
  through bevy_egui needs a per-attempt `add_image` round-trip;
  it's a clean ~50-line follow-up. Listed in `NEXT.md`.
* **Persistent conversation across window close.** Closing the
  composer drops the in-memory history. Saving the conversation
  to a `.cache/maquette/composer-sessions/<datetime>.json` would
  let the user resume; not in scope for v0.10 C-3.
* **LLM-assisted prompt rewriting.** The user picked
  `simple_history` in the design Q&A, so "再做旧一点 → LLM 改
  prompt" is out of scope. Adding it later is a question of
  routing the user's natural-language tweak through one more
  rustyme task before the next Generate.
* **Cross-window state sharing.** The main editor's
  `model_description` / palette are *not* visible inside the
  composer. The composer is a self-contained block-author
  workshop; piping the editor's data in mostly clutters the UI.
* **Per-cell shape vs per-slot shape.** The composer picks one
  shape for the whole "preview run"; the resulting BlockMeta
  carries `shape_hint` consumed by future per-slot binding (D-1).
