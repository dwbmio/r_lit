## What shipped

### A · T-shape multi-angle preview

The original product vision called for a "T-shape, multi-face
preview" visible at all times while painting. v0.7 shipped only the
single PanOrbit perspective view. v0.8 adds **three orthographic
picture-in-picture viewports** — Top / Front / Side — anchored to
the bottom-right corner of the main window:

* Each PIP is a dedicated `Camera3d` with an `OrthographicProjection`
  (`ScalingMode::FixedVertical { viewport_height: 14.0 }`). The
  three cameras share the scene — meshes, outlines, lights — with
  the main perspective camera, so everything you paint shows up in
  all four views at once.
* Viewports are sized in logical pixels (default 180 × 180) with
  DPI scaling applied each frame. On window resize they reflow to
  stay pinned to the bottom-right corner; on small windows they
  gracefully clip but never overlap the status bar (a 32 px
  bottom-reserved strip keeps them clear).
* Small "Top", "Front", "Side" badges are painted by egui in an
  always-on-top layer so the user can identify each panel at a
  glance without needing to guess the angle.
* Toggle via `View → Multi-view Preview`, the **F2** shortcut, or
  the new `Multi` toggle in the floating preview toolbar.

The main perspective camera is untouched — users who disable
multi-view get the v0.7 preview byte-identical. No regression in
the export or the CLI `render` path.

### B · Dock / float preview

The preview can now pop out into an independent OS window — the
other half of the "dockable preview" bullet in the original brief.

* A new `FloatPreviewState` resource toggles the state.
  `FloatWindowPlugin` reacts to the change:
  * **Docked → Floating** — spawns a second `Window` (720×720,
    titled "Maquette Preview") + a `Camera3d` with its own
    `PanOrbitCamera`, initialised to the main camera's current
    yaw / pitch / radius / focus. The second camera is tagged with
    `RenderTarget::Window(WindowRef::Entity(new_window))`.
  * **Floating → Docked** — copies the floating camera's orbit
    state back into the main camera, then despawns the camera and
    the window.
* Closing the floating window via the OS title-bar close button
  goes through the same path: a `WindowClosed` listener flips
  `FloatPreviewState.floating = false` so the next toggle spawns a
  fresh window instead of getting stuck.
* Both windows render the same scene — users with a second monitor
  can paint on the primary window while the floating preview sits
  on the secondary. The main PIPs keep working regardless.

Trade-off: we ship "both windows render simultaneously" rather
than "main goes blank when floating". Less surprising, no extra
clear-color plumbing, and matches the dual-monitor use case a
second window is there to serve.

### C · Empty-state onboarding overlay

A fresh project used to be a blank grid with no affordance. v0.8
paints a centered onboarding panel on any canvas with zero painted
cells:

```
Start painting
  Left-click            paint with the selected color
  Right-click           erase
  1–9                   select a palette color
  Right-click a swatch  edit or delete
```

Implementation: a single `paint_empty_canvas_hint` call inside the
existing `paint_canvas` routine. Drawn via `egui::Painter`, no
extra window / no modal / no dismiss button. Disappears the moment
the user paints their first cell (because `grid.painted_count() >
0`), and only comes back after `File → New` or `Edit → Clear
Canvas`.

### D · Preview QoL — "Fit to Model" + floating toolbar

Two discoverability wins:

* **Fit to Model** — a new `FitPreviewToModel` message handled in
  `camera.rs`. Walks the grid, builds an AABB in world space (same
  origin logic as `preview_mesh`), and sets the PanOrbit camera's
  `target_focus` to the AABB centre + `target_radius` to
  2.4 × half-diagonal (clamped `[3.0, 120.0]`). Orbit angle is
  *not* touched — "fit" means frame, not reset.
  * Exposed via `View → Fit to Model`, the **F** shortcut (standard
    DCC convention), and a "Fit" button in the new preview
    toolbar.
* **Floating preview toolbar** — a small egui `Area` anchored to
  the top-right of the main window with buttons for `Fit`,
  `Reset`, `Multi` (multi-view), and `Float` (dock/float). Sits
  outside all panels so it's always visible regardless of canvas
  scroll or panel sizing. Addresses the v0.7 complaint that the
  status-bar help text was doing discovery work the UI should.

## Under the hood

* New modules, all **GUI-only** (live in `src/`, not in the lib):
  * `src/multiview.rs` — spawns three ortho cameras, keeps their
    viewports in sync with window size + DPI, toggles `is_active`
    from `MultiViewState`, and exposes `pip_logical_rects` so the
    UI can draw labels without re-deriving the geometry.
  * `src/float_window.rs` — spawns and tears down the secondary
    window + camera, copies PanOrbit pose across transitions, and
    listens for `WindowClosed` to stay in sync with the OS.
* `src/camera.rs`
  * Added `MainPreviewCamera` marker so the float-window plugin
    can find the primary camera without a "first one wins" query.
  * Added `FitPreviewToModel` message + `handle_fit_to_model` system.
* `src/ui.rs`
  * New `Fit`, `Reset`, `Multi`, `Float` toolbar anchored top-right.
  * New `paint_empty_canvas_hint` for the onboarding overlay.
  * New `paint_pip_labels` for the Top/Front/Side badges.
  * Menu `View` gains `Fit to Model` and `Multi-view Preview`.
  * Keyboard shortcut block gains **F** and **F2**.
* **No changes** to `lib.rs`, `grid.rs`, `mesher.rs`, `export.rs`,
  `project.rs`, `palette_io.rs`, `render.rs` or
  `src/bin/maquette_cli.rs`. The Headless Invariant is intact:
  `cargo build --no-default-features --bin maquette-cli` still
  succeeds and produces a CLI with no windowing dependencies.

## Decisions locked

1. **T-view layout = picture-in-picture, not splitters.** The
   original brief mentioned a T-shape; the 2×2-splitter reading
   was considered but rejected — splitters add drag handles /
   resize plumbing / per-quadrant state that would push this task
   into v0.9. PIPs in the corner hit the functional requirement
   ("see the model from multiple angles while painting") without
   that cost. If a future user wants resizable quadrants, that's
   an additive v0.9+ change.
2. **PIP viewport height = 14 world units.** Covers a 16×16×8
   canvas comfortably. Larger canvases clip at the edges; this is
   intentional for a glance-only reference view. Revisit if users
   paint much larger shapes routinely.
3. **Float window = one at a time, both render.** Not
   "main-goes-blank-when-floating". Simpler state machine, more
   useful (dual-monitor painting), and the two cameras have
   independent PanOrbit state so the user can pose each
   separately if they want.
4. **Fit factor = 2.4 × half-diagonal, clamped [3, 120].** DCC
   convention; tuned on a couple of test shapes. The clamp
   protects against a single-cell canvas collapsing into a
   near-zero-radius view (unusable pan-orbit).
5. **F / F2 keys live in the GUI only.** They have no equivalent
   in the CLI because there's nothing to "view" headlessly — the
   render verb already picks its own framing. If `render` ever
   needs a "fit" flag it will reuse the same AABB logic from
   `camera::painted_bbox`; that function is currently bin-local
   because it depends on `CELL_SIZE` only, but it could move to
   the lib if a second consumer appears.
6. **Empty-state overlay lives in egui.** Not bevy_ui. Same
   immediate-mode layer as everything else in the canvas panel —
   keeps the onboarding code next to the canvas it documents.

## Known issues / intentionally not shipped

* **No per-window egui UI**. The floating preview window is pure
  3D — no menu, no toolbar inside it. Adding egui to the second
  window would mean a second `EguiContext` and duplicating the
  toolbar, which pays for itself only when the user wants to
  fullscreen the float window. Deferred unless asked.
* **PIPs share input** with the main camera. Dragging in a PIP
  rect will also move the main PanOrbit. The PIPs are
  glance-only — they don't accept mouse input. If users start
  expecting to orbit from a PIP, that's a v0.9 follow-up.
* **No persistence** of `MultiViewState` / `FloatPreviewState`
  across launches. They reset to defaults every cold start. The
  app doesn't have a "user preferences" file yet; v0.9 is the
  right time to add one if it lands alongside autosave.
* **Fit-to-model doesn't reset angle.** "Reset Preview" still
  does a full reset. This is deliberate — conflating the two
  would surprise users who want to frame without losing their
  angle.

## What's still needed to reach v1.0

### v0.9 — robustness

1. **Autosave + crash recovery.** Write a sidecar `.maq.swap` on
   every stroke end; on launch, if a swap exists whose parent
   `.maq` is older, offer to recover.
2. **Packaging.** Bevy's default feature set brings `wgpu` / `naga`
   / `dynamic_linking` we don't need. Trim to the minimum (target:
   < 25 MB release binary). Measured cold-build impact to be
   recorded in `v0.9-complete.md`.
3. **Perf pass.** Profile 32×32×8 canvases — paint latency,
   preview rebuild, PIP render overhead. Budget: 60 fps on an
   M1 base.
4. **Preferences file.** `~/.config/maquette/prefs.toml` for
   viewport toggles (multi-view, float) and recent files list.
5. **Export screenshots.** Archive Godot / Unity / Blender
   round-trip shots under `docs/export/screenshots/` so the
   handoff docs can stop pointing at a TODO.

### v1.0 — release candidate

6. **Docs pass.** README + a short "What is Maquette?" page on
   the repo landing.
7. **App icon + `.desktop` / `.app` metadata.**
8. **Smoke matrix.** macOS / Linux / Windows build + manual-run
   verification checklist.
9. **Release tag** + a versioned `CHANGELOG.md`.

### Out of scope for v1.0

* Splittable multi-view quadrants (PIPs cover the brief).
* FBX export (glTF is the standard, documented).
* "Painting from Top/Front/Side PIPs" — v0.8 ships them as
  reference views only, on purpose.
* Content pipeline (asset library, starter kits) — that's a
  post-v1.0 product-line decision.
