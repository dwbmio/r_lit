# v0.9 A — Autosave + crash recovery · complete

**Shipped 2026-04-23.** Addresses the "crash in the middle of a session
= lose everything" scenario. Edits are continuously flushed to a
sidecar `.maq.swap` as the user paints; on next open of the same
project, a modal offers to recover the unsaved tail.

## Scope

### Delivered

1. **Sidecar swap file** — every project with a `path` gets a
   companion `<path>.maq.swap` that the editor keeps in sync with
   unsaved edits. Swap format = `.maq` format (bit-for-bit), so every
   existing reader (CLI, tests, Godot pipeline, etc.) just works when
   handed a swap path.
2. **Flush triggers** — a swap write happens on:
   - every committed stroke (via
     [`EditHistory::strokes_committed`]'s monotonic counter), and
   - every window-blur event
     (`bevy::window::WindowFocused { focused: false, .. }`).
   Writes are not debounced: strokes are already human-scale, and
   pretty-printed JSON is < 1 ms on SSD for typical projects.
3. **Cleanup** — a successful Save / Save As removes the swap; the
   authoritative `.maq` is now at least as fresh, so any next-session
   modal would be noise.
4. **Recovery modal** — on File → Open, if
   [`maquette::project::swap_is_newer`] says `Some(true)`, the GUI
   renders a centered `egui::Window` asking
   "Recover unsaved changes?" with two deterministic outcomes:
   - **Recover** → load the swap as the project's live state, mark it
     dirty so the user saves explicitly, keep the swap until the next
     manual Save.
   - **Discard** → delete the swap, keep the `.maq` the user opened.
   There is no third state — any choice leaves exactly one
   authoritative file on disk.
5. **CLI integration** — two new integration tests pin the
   Headless Invariant's contract with the sidecar:
   - `cli_reads_swap_file_like_a_regular_project` — a `.maq.swap`
     handed directly to `maquette-cli export` works, no special verb.
   - `cli_export_ignores_sibling_swap_file` — when given a `.maq`
     that has a newer sibling `.maq.swap`, the CLI reads the `.maq`
     it was told about. Protects repeatable CI builds from a stale
     local editor swap.

### Intentionally deferred

* **Untitled projects are not autosaved.** Without a parent path
  there's no natural location for the swap. Full untitled support
  lands in v0.9 C alongside `~/.config/maquette/prefs.toml` (same
  `dirs` dependency buys the untitled-swap anchor).
* **Startup auto-recovery** (no Open needed) also waits on v0.9 C,
  which introduces a last-opened-path persistence. For v0.9 A the
  recovery trigger is File → Open. Users who force-quit and relaunch
  need to re-Open the project — acceptable because the recovery
  modal is the visible contract.

## New public API (lib)

In `maquette::project`:

```rust
pub const SWAP_SUFFIX: &str = ".swap";
pub fn swap_path(project_path: &Path) -> PathBuf;
pub fn swap_is_newer(project_path: &Path) -> Option<bool>;
pub fn write_swap(project_path: &Path, grid: &Grid, palette: &Palette) -> Result<(), ProjectError>;
pub fn remove_swap(project_path: &Path) -> std::io::Result<()>;
```

All four are tested. `swap_path` is a pure function; the other three
are stress-tested against the filesystem with mtime comparisons.

In `maquette::history::EditHistory`:

```rust
pub fn strokes_committed(&self) -> u64;
```

Monotonic counter ticked on each push to the undo stack (including
bare `record` calls). Survives `clear()` so autosave can't be fooled
by File → New.

## New modules (GUI)

* `src/autosave.rs` — the whole flush / recovery policy.
  `AutosavePlugin` registers three Update systems in a `.chain()`:
  `autosave_on_stroke`, `autosave_on_blur`, `cleanup_swap_on_save`.
  Recovery modal lives in `EguiPrimaryContextPass`.
* `src/session.rs` gains an import of `autosave::arm_recovery_prompt`
  and calls it right after a successful Open when
  `project::swap_is_newer` reports `Some(true)`.

## Under the hood

- **Why a counter, not a Message**. `EditHistory` is bin-local today
  but conceptually a pure data structure (its tests run headless).
  Hanging a Bevy `Message<StrokeClosed>` off it would couple the
  data model to the ECS message bus unnecessarily. A `u64` counter
  that autosave polls keeps the layering clean and tested the same
  way the rest of the history is tested.
- **Why monotonic across `clear()`**. On File → New `EditHistory`
  is cleared to drop obsolete undo entries. If the counter reset,
  `AutosaveState::last_flushed_strokes` would look "ahead" of the
  fresh history and the first paint after a New would be missed.
  Monotonicity across clear costs nothing (u64 headroom is vast) and
  removes a subtle race.
- **Why track the project path in `AutosaveState`**. When the user
  switches from `a.maq` to `b.maq` we want the next stroke to flush
  `b.maq.swap`, not falsely trigger on the state difference between
  projects. `state.tracked` matches `current.path` at every tick; any
  mismatch re-baselines without writing.
- **Why re-baseline on Save**. Right after `Save` the `.maq` is as
  fresh as the live state. Without re-baselining, the very next
  stroke tick would immediately flush a swap whose contents are
  identical to the `.maq` we just wrote, wasting an I/O.
- **Why load the `.maq` and then prompt, instead of intercepting
  Open**. The clean intercept-then-prompt design requires a pending
  load queue and an extra resource. Doing the normal load first and
  then offering Recover (which overwrites with the swap) fits the
  existing Bevy systems unchanged; the user sees one frame of the
  `.maq` before the modal appears, which reads as "opened, but wait,
  there's more". This is the cheapest path that's indistinguishable
  from the ideal UX on any modern desktop frame rate.

## Decisions locked

1. **Swap suffix = `.swap`** appended to the full project path.
   `foo.maq` → `foo.maq.swap`. Not `.bak`, not a temp dir, not a
   hidden dotfile. Beside-the-project is visible to the user in
   Finder and greps as `*.maq.swap`.
2. **Flush policy = stroke-committed + window-blur, no timer.**
   Polling-based debouncing adds complexity for no observable win at
   current fixture scale.
3. **Recovery prompt has two answers, no third.** Every outcome ends
   with exactly one authoritative file on disk. No "keep both" —
   collapsed silently because `write_project` is atomic-per-file and
   the swap is a pure intermediate state.
4. **Untitled + startup recovery deferred to v0.9 C.** Not a bug.
   Documented at top of `src/autosave.rs` so anyone touching the
   plugin in future knows it's deliberate.
5. **No toast on each autosave.** Autosave success is invisible.
   Only failure produces a `log::warn!` (not a toast either — users
   shouldn't panic on transient flaky writes). The visible signal is
   crash recovery working on next open.

## Verification debt added by v0.9 A (human — see USER-TODO.md #21)

- Paint a few strokes in a saved project → `kill -9` the process →
  re-open the same path → recovery modal appears → pick Recover →
  edits come back, title bar shows unsaved indicator.
- Same flow, pick Discard → swap is gone, project loads cleanly.
- Save a clean project → verify no `.maq.swap` sidecar is left on
  disk (`ls -la` in the project directory).
- Close the app normally after a Save → no sidecar on disk.
