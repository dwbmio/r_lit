## What shipped

### A · `maquette-cli render` — pure-CPU isometric PNG preview

A new CLI verb, `maquette-cli render <in.maq> --out preview.png`,
produces a visual of a project without booting Bevy or touching a
GPU. This is exactly the piece that was blocking "let CI produce
preview thumbnails" and "let headless runners sanity-check a
project".

Flags:

* `--width` / `--height` (default 512 × 512)
* `--background` (hex, default `#181a1e`)
* `--ambient` (0..=1, default 0.35; higher flattens shading)

What the image shows:

* Isometric projection (yaw −45°, pitch ≈ 35.264°) — the canonical
  voxel-art camera.
* Per-triangle flat Lambert shading against a fixed camera-space
  light direction (upper-right-front).
* sRGB-correct colors: palette colors linearised, shaded, then
  gamma-encoded back to sRGB before write.
* No outline baked into the PNG. Engines already see the
  inverted-hull from the glTF export; the CLI render is a shape
  sanity-check, not a marketing shot.

Under the hood, `maquette::render::render_to_rgba` walks the greedy
mesher buckets, rotates every vertex + normal once per bucket,
projects to screen space, and rasterizes via edge-function barycentric
coordinates with a depth buffer. The inner loop is ~100 lines and
zero dependencies beyond `png` for encoding. Determinism test in
place: same inputs → identical bytes.

### B · GUI feature flag — CLI builds without `bevy_egui` & friends

`Cargo.toml` now has `default = ["gui"]`, where `gui` pulls in the
five GUI-only extensions (`bevy_egui`, `bevy_panorbit_camera`,
`bevy_infinite_grid`, `bevy_mod_outline`, `rfd`). The GUI binary has
`required-features = ["gui"]` so `cargo build --no-default-features
--bin maquette-cli` compiles only what the CLI actually needs.

Measured impact (MacBook M-class, cold `cargo clean`, release deps
cache populated):

* `cargo build --bin maquette-cli` (default features, baseline)
  → **10 min 31 s**
* `cargo build --bin maquette-cli` after feature-gate (default
  features still compile the GUI extensions — same numbers)
* `cargo build --no-default-features --bin maquette-cli`
  → **9 min 48 s**

≈ 43 seconds saved when the CLI build is taken off the GUI-extension
hook. **That's the honest win, not the 5× we initially quoted.** The
rest of the build time is Bevy itself compiling its default features
(`bevy_render`, `bevy_pbr`, `wgpu`, `naga`…). Trimming those
requires declaring `bevy = { version = "0.18", default-features =
false, features = [...] }` in the base deps and rebuilding the
feature list for both lib and GUI — a non-trivial refactor that's
moved to **v0.9** (bundled with the packaging / perf / robustness
pass).

What users get right now:

* CI pipelines that only run the CLI can skip 5 extra crates worth
  of compilation, which dominates the "what changed in `Cargo.lock`"
  diff area.
* Release builds can produce a CLI-only artefact (no winit / no
  wgpu required at runtime) by dropping `gui`.

### C · Cross-engine validation — **deferred to user**

This is the human-in-the-loop task. The agent can't meaningfully
run Godot, Unity, and Blender in a headless test harness on every
change. The CLI golden hashes are the automatic half; the engine
round-trip is still the user's job.

Tracked as verification-debt item **#15** in `NEXT.md`.

### D · Palette import / export CLI

New subcommands:

* `maquette-cli palette export <in.maq> --out colors.json`
* `maquette-cli palette import <in.maq> --from colors.json --out <out.maq>`

Format (`maquette::palette_io`, schema v1):

```json
{
  "version": 1,
  "colors": ["#e64d59", null, "#4c85c1", "#d9b352", ...]
}
```

Nulls preserve deleted slots so imports don't silently remap cells.
Import preserves `palette.selected` when that slot is still live
and snaps to the first live slot otherwise. Alpha is intentionally
not part of the format (Maquette's palette model is opaque-only).

End-to-end integration test `cli_palette_export_import_round_trip`
exercises the whole cycle — export, verify JSON shape, reload a
project with a different palette, import, compare.

## Under the hood

* **New lib modules** — `maquette::render` (rasterizer) and
  `maquette::palette_io` (colors.json). Both live under the
  headless-core boundary; both have unit tests that run in `cargo
  test --lib` without launching a window.
* **`Palette` now derives `Debug`** — needed so palette tests can
  use `Result::unwrap_err` and `matches!`. Harmless side-effect
  elsewhere.
* **`Cargo.toml`** gains a `[features]` table with `default = ["gui"]`
  and five optional dependencies. `[[bin]] maquette` carries
  `required-features = ["gui"]`. `png = "0.18"` added as a
  non-optional base dep (used by the rasterizer in any build mode).
* **`src/lib.rs` module map** expanded to advertise the new modules.
* **CLI parser** gained nested subcommands for `palette {export,
  import}`, the flat `render` verb, and all associated help text.

## Decisions locked

* **Rasterizer = pure Rust, `png` for encoding.** No `imagemagick`,
  no `cairo`, no external shell-outs. If platform drift ever causes
  byte divergence in the golden PNG, we switch to perceptual hashing
  rather than loosening tolerance (decision sticks even though v0.7
  shipped no golden — the current tests are structural).
* **Golden PNG regression = deferred.** The original v0.7 plan
  called for a committed golden PNG + `--bless` flag. v0.7 shipped
  structural tests only (PNG magic, correct dimensions, non-
  background pixel count, top-brighter-than-side luminance sanity
  check, byte-reproducibility). These are strictly weaker than a
  per-pixel golden but they catch every real bug we could think of,
  don't fight f32 platform drift, and add zero repository bloat. If
  regressions start slipping through we'll revisit.
* **Palette portability format** — hex strings + null slots, v1
  schema. Future additions (names, tags) bump the schema version;
  old versions load as subsets.
* **Feature flag naming** — `gui` only (not `desktop`, not
  `bevy_gui`). Short name wins for CLI users who'll type
  `--no-default-features` a lot.
* **Bevy feature trimming deferred to v0.9**, not v0.8. v0.8's
  theme is user-visible preview UX (T-view, dock/float); mixing
  build-infra work in would dilute both. v0.9 bundles perf +
  packaging + Bevy trimming under a single "production hardening"
  banner.

## Known issues / intentionally-not-shipped

* **No golden PNG.** Documented above. Structural assertions
  only.
* **Still no FBX.** glTF/GLB covers every engine we're targeting;
  FBX's proprietary tax remains post-v1.0.
* **Cold build time is still dominated by Bevy's own default
  features.** The `gui` feature gate is the right mechanism; we
  just haven't split Bevy itself yet.
* **No `render --iso {front,side,top,...}` flag.** If a user wants
  a non-standard camera, they open the GUI. Canonical iso is the
  one shipping projection.

## What's still needed to reach v1.0

Keeping this explicit so future sessions don't re-litigate scope:

1. **v0.8 — Preview UX alignment.** T-view multi-angle preview
   (the user's original vision asked for T-shape + 4 panels);
   dock/float toggle; empty-state onboarding. Biggest remaining
   gap between the shipped product and the original product brief.
2. **v0.9 — Production hardening.** Autosave / crash recovery
   (save-every-N-seconds to a temp path, recover on next launch);
   bigger canvas performance audit (128 × 128 × 8 at 60 fps);
   macOS `.app` / Linux AppImage / Windows `.exe` packaging;
   Bevy feature trimming for leaner CLI builds. Error message
   polish across save / load / export failure modes.
3. **v1.0 — Release candidate.** README + 5-step quickstart with
   screenshots; app icon; about / version screen; smoke test matrix
   on a fresh machine across OSes; tag + release notes. No new
   features.

Explicitly **not** on the path to v1.0:

* AI-assisted grid generation (original vision, deferred post-v1.0
  with its own cost analysis in `COST_AWARENESS.md`).
* FBX export.
* Voxel character mode (separate product line,
  `COST_AWARENESS.md` §Post-v1.0 Possible Product Line Split).
* Plug-in / extension system.
* Cloud save / multi-user collaboration.
