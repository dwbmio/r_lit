# v0.10 C-1 — Project schema v4 (lib only)

**Shipped 2026-04-27 (late evening).** Lib-side data-model work for
the AI texture pipeline. No worker dependency. Lays the groundwork
for D-1 ("one-shot full-model textures") by adding the project file
fields, the `Palette` slot-meta plumbing, and the headless
load/save APIs that D-1 will hook into.

The GUI binary's actual *editing* of these fields (text input,
undo-stack integration, autosave plumbing) is intentionally
deferred to **v0.10 C-2 / D-1**, where it will land alongside the
GUI Generate button.

## §1 — What landed

### 1.1 New module: `maquette::texture_meta` (`src/texture_meta.rs`)

Three plain-data types that live in the headless lib so the GUI,
the CLI, and the project file format all agree on shape:

* `TextureHandle { cache_key: String, generated_at: i64 }` — points
  into the existing `~/.cache/maquette/textures/<sha>.png` store
  owned by [`crate::texgen`]. Bytes never live on a `TextureHandle`;
  resolution goes through `texgen::cache_get`.
* `PaletteSlotMeta { override_hint: Option<String>, texture:
  Option<TextureHandle> }` — per-slot data. `is_empty()` returns
  `true` when both fields are `None`; that's the case where the
  v4 serialiser collapses the slot to `null` to keep `.maq` files
  visually compact.
* `TexturePrefs { view_mode: PaletteViewMode, ignore_color_hint:
  bool }`. `PaletteViewMode = Flat | Textured`, default `Flat`.

Every field is `#[serde(default)]` and the structs use
`skip_serializing_if = "Option::is_none"` so older builds and
hand-edits round-trip cleanly.

### 1.2 `grid::Palette` gained `slot_meta`

```rust
pub struct Palette {
    pub colors: Vec<Option<Color>>,
    pub selected: u8,
    pub slot_meta: Vec<PaletteSlotMeta>,   // ← new
}
```

`slot_meta.len() == colors.len()` is an invariant maintained by:

* **`Palette::default()`** — initial palette gets 12 default meta
  entries.
* **`Palette::from_colors(colors, selected)`** — new helper for
  tests / loaders that already have a sparse `Vec<Option<Color>>`.
  Pads `slot_meta` to `colors.len()` automatically.
* **`Palette::add(color)`** — pushes a default meta when growing
  the vector; on vacant-slot reuse, *resets* the meta so a freshly
  added color never silently inherits the previous (deleted)
  slot's `override_hint` / `TextureHandle`.
* **`Palette::delete(idx, ...)`** — clears the meta at `idx`
  alongside the color. Same reasoning: future re-add lands in a
  clean slot.
* **`Palette::ensure_meta_alignment()`** — idempotent realignment.
  The project loader calls it after deserialising a malformed /
  truncated `palette_meta` so a length mismatch never blocks file
  recovery (it's logged as `debug`, not propagated as an error).

New typed mutators that double as undo-stack hooks for D-1:

* `set_override_hint(idx, Option<String>) -> Option<Option<String>>`
  — returns the previous value, normalises whitespace-only input
  to `None`.
* `set_texture(idx, Option<TextureHandle>) -> Option<Option<TextureHandle>>`
  — same shape.
* `meta(idx)` / `meta_mut(idx)` accessors.

`palette_io::import_palette_into` now calls
`ensure_meta_alignment` after replacing `palette.colors`, so an
imported `colors.json` (which has no meta concept) restores the
invariant immediately.

### 1.3 `project.rs` schema v3 → v4

`SCHEMA_VERSION = 4`. Three new top-level optional fields:

```json
{
  "version": 4,
  "grid": { … },
  "selected_color": 3,
  "palette": [ … ],

  // v4 additions, all #[serde(default)]:
  "model_description": "minecraft-style grass dirt block",
  "texture_prefs": { "view_mode": "flat", "ignore_color_hint": false },
  "palette_meta": [
    { "override_hint": "patchy moss top",
      "texture": { "cache_key": "deadbeef…", "generated_at": 1700000000 } },
    null,
    {}
  ]
}
```

Backward compat:

* **v1, v2, v3 files** load unchanged. Missing v4 fields hit their
  defaults (`""` / `TexturePrefs::default()` / all-default
  `slot_meta` sized to match the loaded palette).
* **v4 files written by hand or future builds** with
  `palette_meta.len() != palette.len()` are accepted: the loader
  pads or truncates with default meta and logs a debug line. Same
  for `null` entries inside `palette_meta` — they decode as
  `PaletteSlotMeta::default()`.

Forward compat:

* Defaults are skip-serialised: a project that never touches the
  AI texture features (empty description, default prefs, all
  empty meta) writes the same compact JSON it always did, plus a
  `palette_meta: [null, null, …]` array. Hand-inspecting the file
  isn't noisier than v3.

### 1.4 New top-level type: `project::ProjectMeta`

```rust
pub struct ProjectMeta {
    pub model_description: String,
    pub texture_prefs: TexturePrefs,
}
```

Lives next to `Grid` + `Palette` rather than inside `Palette`
because both fields are project-wide. The GUI will register this
as `Resource<ProjectMeta>` in D-1.

### 1.5 New API surface (additive only)

```rust
// Old API preserved verbatim — drops project meta on the floor:
pub fn read_project(&Path) -> Result<(Grid, Palette), ProjectError>
pub fn write_project(&Path, &Grid, &Palette) -> Result<(), ProjectError>
pub fn apply_to_grid_and_palette(&Path, &mut Grid, &mut Palette) -> ...

// New v0.10 C API — preserves ProjectMeta:
pub fn read_project_with_meta(&Path)
    -> Result<(Grid, Palette, ProjectMeta), ProjectError>
pub fn write_project_with_meta(&Path, &Grid, &Palette, &ProjectMeta)
    -> Result<(), ProjectError>
pub fn apply_to_grid_palette_meta(&Path, &mut Grid, &mut Palette,
                                   &mut ProjectMeta) -> ...
```

The CLI (`maquette-cli`), `session.rs`, and `autosave.rs` are
unchanged in C-1 — they continue to use the legacy entry points.
That's deliberate: v3 → v4 round-trip via the legacy APIs
produces a v4-shaped file with default meta, which is exactly the
behaviour `#TEX-C` asks for. Migration of those code paths to the
`*_with_meta` variants is part of **C-2 / D-1** when the GUI
actually starts editing description / prefs and an autosave write
that drops them would be visible.

## §3 — What's deliberately *not* in C-1

### EditHistory integration (deferred to D-1)

`history.rs` lives in the GUI binary, not in the lib. The natural
shape — extending `PaintOp` (or wrapping it in an `EditOp` enum
including `SetModelDescription` / `SetOverrideHint`) — also
requires `handle_history_action` to gain access to
`ResMut<Palette>` and `ResMut<ProjectMeta>`, which are GUI
resources. Doing it in C-1 would have:

1. forced an early decision on the system-set ordering of the
   undo plugin vs. the AI generate plugin (which doesn't exist
   yet), and
2. introduced a code path that's exercised only by the GUI, so
   none of the new lib tests would actually cover it.

Lib-side, `Palette::set_override_hint` / `set_texture` already
return the previous value, so D-1 has a clean hook to push
undo entries onto whichever event type lands. NEXT.md will pick
this up.

### CLI surface (deferred to D-1 or later)

`maquette-cli info` doesn't print `model_description` or texture
counts yet. The CLI is intentionally narrow until the user-facing
phrasing settles — D-1 is the better moment.

### Autosave migration (deferred to D-1)

`autosave.rs` calls `write_swap` → `write_project`, which in C-1
silently writes default `ProjectMeta`. Right now that's harmless
(no GUI surface lets the user *set* description yet), but D-1
must migrate autosave to `write_project_with_meta` before any
description-editing UI ships, otherwise an autosave between two
keystrokes could erase a typed description. Tracked in
`NEXT.md`.

## §4 — Public API changes

### Added

* `pub mod maquette::texture_meta` (whole module).
* `pub fn maquette::grid::Palette::from_colors(...)`.
* `pub fn maquette::grid::Palette::meta` /
  `meta_mut` / `set_override_hint` / `set_texture` /
  `ensure_meta_alignment`.
* `pub struct maquette::grid::Palette { ..., slot_meta: Vec<…> }`
  — new public field. Code that pattern-matches the struct
  literally needs to add `slot_meta`; code that reads
  `palette.colors` is unchanged.
* `pub struct maquette::project::ProjectMeta`.
* `pub fn maquette::project::read_project_with_meta`.
* `pub fn maquette::project::write_project_with_meta`.
* `pub fn maquette::project::apply_to_grid_palette_meta`.

### Changed semantics (no signature change)

* `read_project` / `apply_to_grid_and_palette` — drop project meta
  silently. Document'ed in their doc comments.
* `write_project` / `write_swap` — write a v4-shaped file with
  default project meta. Idempotent for projects whose `ProjectMeta`
  is already default; lossy for projects whose meta was set in
  memory but isn't being passed through the legacy API. C-2 fixes
  the lossy case.

### Removed

Nothing. Strictly additive change set.

## §6 — What's next

The next agent task is **D-1**: GUI material panel +
Canvas-group fan-out, which finally puts the texture pipeline in
front of the user. D-1 also picks up the loose ends listed in §3:

1. Migrate `autosave.rs` + `session.rs` to the `*_with_meta`
   APIs so the new `ProjectMeta` resource survives autosaves
   and Save As.
2. Extend `EditHistory` (in `bin/history.rs`) to a
   `record_op(EditOp)` API where `EditOp` is `Paint(PaintOp) |
   SetModelDescription { … } | SetOverrideHint { … }`. Wire
   `Palette::set_override_hint` / `set_texture` (which already
   return the previous value) into that path.
3. Add a `Canvas group` fan-out check on the worker side
   (sonargrid Stage 3.5 from the roadmap) — the only piece of
   D-1 that needs cross-team verification.

`USER-TODO.md` `#TEX-C` is now **all the lib-side work done**.
The "open v3 → save → re-open" check it describes can already be
run today against a CLI build:

```sh
maquette-cli info my_v3_project.maq               # opens fine
maquette-cli palette export my_v3_project.maq …   # works
# (Save through the GUI to confirm version=4 lands on disk.)
```

What remains for the human-eyes pass is the GUI side of #TEX-C
(the "override_hint edit enters undo chain" clause), which only
exists once D-1 ships an editor for it.
