## ¬ß1 ‚Äî Why a B-bis instead of just a follow-up commit

The original v0.10 B `texgen.rs` worked exactly as specified, but
the **specification was hypothetical**. By the time we plugged in
the live sonargrid we found:

1. The deployed task name is `texgen.gen`, not `texture.gen`.
2. Two queues are deployed (`rustyme:texgen-cpu:*` for free
   programmatic + LLM-smart synthesis, `rustyme:texgen-fal:*` for
   Fal.ai FLUX schnell) instead of the single `rustyme:texgen:*`
   the roadmap planned.
3. The result envelope grew a multi-format shape
   (`{image_b64, format, content_type}`) on top of the legacy
   `{png_b64}` ‚Äî workers emit both when `format == "png"`.
4. New `kwargs` exist (`output_format`, `jpeg_quality`,
   `style_mode`).

Plus one **producer-side bug** that only manifests against a
result list with stale entries ‚Äî see ¬ß3.

Calling it a B-bis (rather than rolling these into "C") keeps
`#TEX-B` cleanly checkable on USER-TODO and preserves the
"worker shape changes need a deliberate doc bump" discipline.

## ¬ß2 ‚Äî Wire-shape alignment

| Item | v0.10 B (planned) | v0.10 B-bis (actual sonargrid) |
|---|---|---|
| `task` name | `texture.gen` | **`texgen.gen`** |
| queue key (default) | `rustyme:texgen:queue` | **`rustyme:texgen-cpu:queue`** |
| result key (default) | `rustyme:texgen:result` | **`rustyme:texgen-cpu:result`** |
| result PNG field | `png_b64` | **`image_b64` + `format` + `content_type`** (legacy `png_b64` still accepted) |
| Format support | PNG only (implicit) | PNG required at the producer (explicit + actionable error if `format != "png"`) |

### Code changes (`src/texgen.rs`)

* `RustymeConfig::from_env`:
  * Default `task_name` ‚Üí `"texgen.gen"`.
  * New env `MAQUETTE_RUSTYME_PROFILE` (values: `cpu` / `fal`,
    default `cpu`) ‚Äî shorthand that resolves to the right
    `*_KEY` pair. Accepts the long forms `texgen-cpu` /
    `texgen-fal` too. Explicit `MAQUETTE_RUSTYME_QUEUE_KEY` /
    `MAQUETTE_RUSTYME_RESULT_KEY` still wins.
  * New env `MAQUETTE_RUSTYME_STYLE_MODE` (`auto` / `solid` /
    `smart`, omitted ‚Üí worker default). Only consumed by
    `texgen-cpu`; ignored by `texgen-fal`.
* New `pub enum RustymeProfile { Cpu, Fal }` plus `parse() /
  queue_key() / result_key() / admin_name()` ‚Äî single source of
  truth for the deployed naming, exercised by the tests so a
  drift here fails CI.
* `kwargs` payload now pins `"output_format": "png"` so the
  smart Lua hook can't autonomously return JPEG bytes (which
  it does for "toon base color" / "metallic surface" prompts ‚Äî
  it correctly identifies them as alpha-free, but Maquette's
  cache stores `<sha>.png` and the v0.10 D GUI decoder
  assumes PNG).
* `TextureResult` deserialiser:
  * Both `image_b64` and `png_b64` are now `Option<String>` so
    we can emit a custom error when neither is present, instead
    of a generic serde "missing field" failure.
  * `decode_png_b64()` resolves the bytes string with explicit
    precedence: prefer `image_b64` when `format == "png"`,
    fall back to `png_b64`, error on every other shape with a
    message that names both accepted forms.
* Doc-comment on `pub mod rustyme` rewritten to reflect the
  real worker contract; `docs/texture/rustyme.md` rewritten end
  to end (smoke test now points at the live sonargrid host,
  env-var table updated, profile/style-mode rows added).

## ¬ß3 ‚Äî Producer bug fix: foreign-reply infinite loop

While verifying the round-trip we hit a non-deterministic timeout
that turned out to be a real bug present since v0.10 B (and also
present in `rustyme-py`'s Python SDK):

**Symptom.** First `texture gen --provider rustyme` against a
real cluster timed out after 60s. The worker had **succeeded**
(its reply with our task_id was visible in the result list), but
Maquette never saw it.

**Cause.** Workers `LPUSH` results to the head of the list;
Maquette `BRPOP`s from the tail (FIFO). When a foreign reply
(another producer's task_id) arrives at the tail, v0.10 B did
`RPUSH` to put it back ‚Äî but RPUSH writes to the tail, where
BRPOP is reading from, so the same foreign reply got pulled and
re-pushed forever. Our reply was *one slot behind it*, blocked
by the cycle.

```
worker:    LPUSH result   ‚îÄ‚îÄ‚ñ∂  [our reply][foreign1][foreign2]‚Ä¶[foreignN]
                                ‚Üë head                          ‚Üë tail
producer:  BRPOP result   ‚óÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
                                                                    ‚Üª (RPUSH back to same spot ‚Äî deadlock)
```

In the wild this only manifests when the result list contains
**stale entries that were never drained** (e.g. a previous
client crashed mid-await, or ‚Äî as today ‚Äî the cluster has been
running test traffic). On a fresh cluster the bug is invisible.

**Fix.** Foreign replies (and unparseable payloads) are now
`LPUSH`-ed back to the **head**, matching the worker's own write
direction. Subsequent BRPOPs walk older elements until ours
surfaces. We logged a flag with the sonargrid maintainer so
`rustyme-py` can take the same patch (`AsyncResult.get` line 92).

```
loop iteration N:                    [foreignK]‚Ä¶[our reply]‚Ä¶[foreignK-1]
                                      ‚Üë head, just LPUSH'ed back        ‚Üë tail, next BRPOP's target
```

Verified by the live smoke run: with 5 stale FAILURE entries
already in `rustyme:texgen-cpu:result`, the producer completed
in ~2s instead of timing out at 60s.

## ¬ß5 ‚Äî Public API changes

* `pub enum maquette::texgen::rustyme::RustymeProfile { Cpu, Fal }`
  with `parse() / queue_key() / result_key() / admin_name()`.
* `RustymeConfig` gained:
  * `style_mode: Option<String>` field;
  * default `task_name = "texgen.gen"`;
  * default `queue_key = "rustyme:texgen-cpu:queue"`;
  * default `result_key = "rustyme:texgen-cpu:result"`.
* All other public surface unchanged.

Old call sites that constructed `RustymeConfig` by literal need
to add `style_mode: None`. `RustymeConfig::from_env()` callers
(the only path the CLI uses) get the new defaults transparently.

## ¬ß7 ‚Äî Known caveats / follow-ups

* **`style_mode` is not in the disk cache key.** A request with
  the same `prompt+seed+w+h+model` returns identical cached
  bytes whether you ask for `solid` or `smart`. This is fine
  for production (you typically pin one mode per Maquette
  install) but if you're A/B comparing modes interactively, pass
  `--no-cache`. Documented in `docs/texture/rustyme.md`.
* **Multi-format consumption (JPEG / WebP) deferred.** The
  worker is willing; the GUI decoder + cache-filename scheme
  need to learn `content_type` first. Tracked as a follow-up
  in the wire-protocol doc; not on the v1.0 critical path.
* **`rustyme-py` SDK still has the foreign-reply RPUSH bug.**
  Maquette is correct; the upstream Python client uses the
  same naive RPUSH and will deadlock under the same
  conditions. Flagged for the sonargrid maintainer.
