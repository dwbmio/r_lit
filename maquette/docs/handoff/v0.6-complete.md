# v0.6 — palette editor + stroke undo + greedy meshing

**Status:** shipped 2026-04-23. Three independent feature lines, all
with headless unit coverage and at least one CLI-invocable
integration test where applicable.

The original v0.6 plan had four sub-tasks A/B/C/D. D (headless
`maquette-cli render`) is stretch; shipping it responsibly needs a
pure-CPU rasterizer or a Bevy-headless pipeline, neither of which
should be rushed. It has been demoted to **v0.7 A**. Everything else
in the v0.6 roadmap landed.

## User-visible

### Palette editor (sub-task A)

- **Right-click any swatch** in the palette bar. A context menu
  opens with:
  - An inline color picker (egui's `color_edit_button_rgb`). Edits
    update the palette live; the preview mesh re-colors on the
    next frame.
  - A `Delete…` button that opens a confirmation modal.
- **Delete-color modal** shows the slot number, a color preview,
  and the count of painted cells currently using the color. The
  user picks one of:
  - *Erase* — those cells become empty. Default.
  - *Remap to:* — pick another live palette index from a dropdown;
    all cells using the deleted color re-paint to the remap
    target. Disabled when no other live colors exist.
- **"+" slot** at the end of the palette bar adds a new color.
  Default hue-shifted 45° from the currently-selected color
  (`Hsla::from(base)`), which keeps the new swatch in the same
  color family but visually distinguishable. The new slot is
  auto-selected so the user can paint with it immediately.
- **1-9 keyboard shortcuts** now target the nth *live* color.
  Deleted slots don't leave dead shortcuts behind.

### Stroke grouping in undo history (sub-task B)

- **Drag across multiple cells with the paint tool and press
  Ctrl+Z once** — the entire stroke rolls back, not just the
  most recently painted cell.
- Same for right-click-drag erase.
- Single clicks (no drag) still produce one undo entry, same as
  before.
- `Redo` replays the full stroke forward.

### Greedy rectangle meshing (sub-task C)

- **Exports are smaller.** A 16×16 flat slab now emits 6 body
  quads instead of 48. A 1×1×8 column emits 6 quads instead of
  34. Proportional savings on file size, preview draw weight, and
  engine import time.
- **Preview and export stay perfectly in sync.** Both call
  `maquette::mesher::build_color_buckets`, which is now greedy
  by default. A CLI integration test asserts the export path
  uses the greedy mesher — if anyone accidentally wires to the
  culled mesher, CI fails.
- **Surface is unchanged.** A regression oracle
  (`build_color_buckets_culled`) is retained for tests;
  equivalence tests assert the greedy output covers the same
  exterior surface area with strictly-fewer-or-equal quads.

## Under the hood

### Palette refactor (`grid.rs`)

- `Palette::colors` is now `Vec<Option<Color>>`. `None` means
  "this slot was deleted and is available for reuse". This is
  the single decision that makes index stability feasible: every
  non-deleted color keeps its original `u8` index for the life of
  the project, no matter how many delete / re-add cycles happen
  around it.
- New public API:
  - `get(idx) -> Option<Color>`
  - `is_live(idx) -> bool`
  - `live_count() -> usize`
  - `iter_live() -> impl Iterator<Item = (u8, Color)>`
  - `add(color) -> Option<u8>` — reuses the first `None` slot
    before growing; returns `None` iff `MAX_PALETTE_SLOTS (256)`
    is already full.
  - `update(idx, color) -> bool` — only succeeds on live slots.
  - `delete(idx, grid, mode) -> bool` — updates the grid first,
    then clears the slot. Invalid remap targets fall back to
    erase.
  - `usage_count(grid, idx) -> usize` — for the "N cells affected"
    text in the delete modal.
- `DeleteColorMode` enum: `Erase | Remap { to }`.
- 9 unit tests in `grid::palette_tests` cover add / update /
  delete / selection snap / usage count; all reachable from
  headless `cargo test`.

### Project file schema v3 (`project.rs`)

- Palette field is `Option<Vec<Option<RgbaPayload>>>`. Deleted
  slots serialize as `null`; live slots serialize as RGBA
  objects.
- v1 files (no palette) and v2 files (dense palette) still load
  unchanged: serde deserializes `Vec<Option<T>>` from a no-nulls
  JSON array the same as `Vec<T>`, so no migration code is
  needed.
- On load, `selected_color` is clamped to a live slot — a saved
  file that references a deleted slot (possible after a round
  trip through older buggy builds) snaps to the first live
  color instead of producing a UI with a dead selection.
- Three new tests exercise v2 compatibility, v3 round-trip with
  a deleted slot, and dead-selection fallback.

### Stroke history (`src/history.rs`)

- `EditHistory.undo` / `.redo` are now `VecDeque<Stroke>`.
- `Stroke` wraps `Vec<PaintOp>`. One stroke = one pointer-down
  to pointer-up gesture.
- `begin_stroke()` / `end_stroke()` bracket the gesture;
  `record(op)` appends to the open stroke or falls back to a
  single-op stroke when no gesture is open (clicks, keyboard
  shortcuts, non-UI callers).
- `end_stroke` silently discards empty strokes, so a click that
  doesn't actually change anything (e.g. paint-over-same-color)
  doesn't pollute the history.
- `handle_history_action` calls `end_stroke` before each undo /
  redo: if the user hits Ctrl+Z mid-drag (rare but possible),
  the in-flight stroke is committed first so undo reaches a
  well-defined snapshot.
- 9 unit tests drive `EditHistory` directly — no Bevy App, no
  UI, pure data-structure correctness. Satisfies the Headless
  Invariant even though the type lives in the GUI binary.

### Greedy mesher (`src/mesher.rs`)

- Classic 2D greedy meshing, one pass per face direction:
  - For each slab perpendicular to the face axis, build a 2D
    mask of the visible face colors at that slab.
  - Run a greedy rectangle merger: walk the `u` axis keeping
    a strip of matching cells in `v`, extend the strip in `u`
    as long as the whole `v`-range matches.
  - Emit one quad per merged rectangle with the correct CCW
    winding for the face direction.
- Per-color bucketing preserved: same-color merges across voxel
  boundaries, different colors stay in separate `MeshBuilder`s.
- `build_color_buckets_culled` kept as the regression oracle.
- 8 greedy-mesher unit tests + 4 retained culled tests + 1 CLI
  integration test (`cli_export_uses_greedy_mesher`) asserting
  the exported GLB's triangle count matches the greedy output
  and is strictly smaller than the culled equivalent.

## Decisions locked

1. **Palette is sparse.** Deleted slots stay as `None`; the Vec
   never shifts elements. Saved projects preserve every non-
   deleted slot's index across edits, across sessions, across
   collaborators.
2. **Delete fallback is erase by default.** Remap is the user's
   explicit choice. If the user picks remap but the target slot
   is invalid (out-of-bounds / deleted / same as the deleted
   slot), the operation falls back to erase rather than leaving
   dangling references.
3. **Stroke granularity = pointer-stream.** One pointer-down to
   pointer-up event is one undo entry, regardless of how many
   cells it touches. Dragging off-canvas and re-entering is
   still one stroke (egui keeps the drag alive).
4. **Greedy is the default, culled is the oracle.** No feature
   flag, no user-visible toggle. The old culled mesher exists
   only as test scaffolding; a CLI test guards against anyone
   accidentally routing shipping code to it.
5. **Palette schema version 3.** Bumped because the in-memory
   shape (`Vec<Option<Color>>`) is a material change, even
   though the deserializer stays backward-compatible with v1
   and v2.

## Known issues / deliberately deferred

- **CLI `render` PNG preview** — deferred to v0.7 A. Adds a
  pure-CPU rasterizer and a golden-file visual-regression test.
- **Bevy feature-gating** — the CLI binary still pulls the
  whole Bevy render dependency tree. Splitting the GUI behind a
  `gui` feature is v0.7 B; build-time win estimated ~5×.
- **Palette import / export** — a separate `.palette.json` file
  that users can share / re-apply across projects. Candidate
  for v0.7 stretch.
- **Color edit history** — right-click color edits do not
  currently land in `EditHistory`. Only paint / erase are
  undoable. If this matters in practice we'll add a
  `HistoryEntry` variant for palette ops; no user has asked.
